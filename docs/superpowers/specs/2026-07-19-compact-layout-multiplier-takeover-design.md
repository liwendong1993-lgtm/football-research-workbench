# 足球研究工作台紧凑布局与倍投功能接管设计

## 背景

Hermes 已在 `football-research-workbench` 中实现了本轮需求的大部分代码，但其代理运行时在处理结构化 assistant content 时崩溃，导致浏览器验证、缓存版本更新、提交和部署没有完成。本次接管以现有未提交改动为基础，完成验证、必要修正和发布，不重新设计整个应用。

## 目标

1. 保留现有深色视觉体系，只借鉴参考比分 App 的高密度信息布局。
2. 将首页顶部研究概览压缩为单行，减少首屏占用。
3. 将比赛卡片压缩为两行，使手机首屏能展示更多比赛，同时不丢失赔率和研究摘要。
4. 为组合方案增加投注倍数，并让投注金额、理论最低返奖、理论最高返奖及生成图片同步变化。
5. 保持此前完成的点选自动保存和夸克图片弹窗兼容能力。
6. 完成自动化测试、浏览器交互验证、静态缓存更新、提交和 GitHub Pages 部署。

## 非目标

- 不改成参考截图中的白色主题。
- 不改变比赛数据来源、组合算法的玩法规则或本地优先的数据架构。
- 不增加账号、云同步、支付、代购或投注功能。
- 不进行与本轮需求无关的大规模重构。

## 交互与布局设计

### 顶部概览

顶部概览保持深色卡片和金色强调，内容收敛为一行：

- 数据同步状态；
- 当前研究日期；
- `已研究 X/Y` 进度。

移除占用较大的环形进度和重复副标题。窄屏时保持单行，优先压缩间距和字号，不允许产生横向页面溢出。

### 比赛卡片

每张比赛卡片使用两行结构：

1. 第一行：联赛、比赛编号、主队、`VS`、客队、开赛时间。
2. 第二行：胜平负赔率、信心及已选玩法摘要。

主客队名称允许省略号截断，并通过 `title` 保留完整名称。研究标签过多时只在标签区域内部横向滚动，不扩大卡片高度，也不造成页面整体横向滚动。深色背景、边框、主推金色标识和现有点击进入编辑行为保持不变。

### 投注倍数

组合方案新增整数倍数，合法范围为 1–99999，默认值为 1。倍数应：

- 在新建或编辑组合时保存；
- 在方案卡片中可直接修改并即时持久化；
- 同比例放大投注金额、理论最低返奖和理论最高返奖；
- 出现在单方案海报和详细研究长图中；
- 对旧数据兼容：缺少倍数字段时按 1 倍处理。

空值在用户继续输入时不强制改写；失焦或保存时归一化为合法整数。小数向下取整，非法值回退为 1，超出范围的值截断到边界。

## 兼容性与错误处理

- 点选比赛玩法、信心、比分或修改分析文字后继续自动保存，无需滚动到底部点击保存。
- 海报弹窗继续使用普通固定定位容器，不恢复夸克存在触控问题的原生 `dialog`。
- PNG 图片预览加载失败时隐藏破图并回退 Canvas。
- 支持时优先调用 Android 系统分享；不支持时保留 Blob 下载、打开大图及长按保存路径。
- Local Storage 写入失败时在当前交互区域显示明确错误，不假装保存成功。
- 发布时提升 HTML、JS、CSS 和 Service Worker 的统一资源版本，避免夸克继续命中旧缓存。

## 测试与验收

### 自动化测试

- 1 倍和多倍方案的金额及返奖同比例变化。
- 非法、空、小数和超范围倍数的归一化。
- 旧组合数据缺少倍数时按 1 倍计算。
- 顶部概览为单行结构。
- 比赛卡片为两行紧凑结构，且保留赔率与研究摘要。
- 自动保存行为仍存在。
- 夸克弹窗不使用原生 `dialog`，图片失败具备 Canvas 回退。
- 所有既有测试继续通过。

### 浏览器验证

- 360px、390px、430px 手机视口：无页面横向溢出，首屏比赛密度明显提高。
- 桌面视口：卡片仍清晰，不因手机压缩规则产生异常空白或错位。
- 实际点击验证：打开比赛、选择玩法、关闭重开确认自动保存；创建组合、修改倍数、刷新确认持久化；打开海报并验证按钮可点击。
- 检查浏览器控制台无新增错误，并验证 Service Worker 使用新缓存版本。

## 发布策略

1. 保留并审查 Hermes 的未提交改动。
2. 对发现的缺口先补失败测试，再做最小修复。
3. 运行完整自动化测试和浏览器验收。
4. 更新静态资源及 Service Worker 缓存版本。
5. 将功能改动作为独立提交推送到 `origin/main`，由现有 GitHub Pages 流程发布。
6. 发布后访问线上地址进行一次冒烟检查。

## 完成标准

- 本文目标全部满足；
- 自动化测试全部通过；
- 目标手机与桌面视口验证通过；
- 未破坏自动保存和夸克兼容路径；
- 工作树功能改动已提交并推送；
- 线上页面加载新资源且核心交互可用。
